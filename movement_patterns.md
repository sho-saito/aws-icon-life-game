# AWS Icon Life - アイコン移動パターン

このドキュメントでは、AWS Icon Lifeゲーム内での各AWSサービスアイコンの特徴的な移動パターンについて説明します。各サービスの特性や役割に基づいた動きを実装することで、より直感的にサービスの性質を理解できるようになっています。

## 実装済みの移動パターン

### EC2 (Elastic Compute Cloud)

EC2は、AWSの主要なコンピューティングサービスとして、比較的直線的かつ目的志向の動きをします。また、VPCへの依存関係を視覚的に表現するため、VPCに引き寄せられる特性を持ちます。

**特徴:**
- **直線的な動き**: 方向転換の確率が低く（2%）、一度設定した方向に直線的に進む傾向がある
- **VPCへの引力**: VPCが近くにある場合、EC2はVPCに向かって引き寄せられる
- **距離に応じた引力の強さ**:
  - 遠距離（300px以上）: 強い引力（0.15）
  - 中距離（100px〜300px）: 中程度の引力（0.08）
  - 近距離（100px未満）: 引力なし（すでに十分近い）

**実装の意図:**
- EC2インスタンスはVPC内に存在する必要があるという依存関係を視覚的に表現
- コンピューティングリソースとしての「目的志向」の性質を直線的な動きで表現
- VPCから離れすぎると「不安定」になる様子を引力で表現

### S3 (Simple Storage Service)

S3は、AWSの主要なストレージサービスとして、安定性と信頼性を重視した動きをします。急激な変化が少なく、ゆっくりと動く特性を持ちます。

**特徴:**
- **低速移動**: 最大速度が通常の1/3（1.0）に制限され、全体的にゆっくり動く
- **ランダムな減速**: 5%の確率で速度が10%減少し、徐々に遅くなる傾向がある
- **緩やかな方向変化**: 10%の確率で少しだけ方向を変える（±0.2ラジアン程度）
- **時々停止する動き**: 1%の確率で完全に停止し、最大2秒後に再始動

**実装の意図:**
- ストレージサービスとしての「安定性」と「信頼性」を遅い動きで表現
- データが静的に保存される性質を、時々停止する動きで表現
- 急激な変化が少ないという特性を、緩やかな方向変化で表現

### VPC (Virtual Private Cloud)

VPCは、AWSのネットワーク基盤として、広い範囲をカバーしながらも安定した動きをします。他のサービスの基盤となる性質を表現するため、円を描くような動きをします。

**特徴:**
- **低速移動**: 最大速度が0.8に制限され、安定したネットワーク基盤を表現
- **円運動**: 毎フレームで一定方向（時計回りまたは反時計回り）に角度を変え、円を描くように移動
- **一定の回転方向**: 各VPCインスタンスごとに回転方向（時計回りまたは反時計回り）が決まる
- **中央への引力**: 画面中央から遠ざかりすぎると中央に引き戻される傾向がある
- **希少性による回復**: マップ上のVPC数が5個以下の場合、体力回復速度が通常の4倍に上昇

**実装の意図:**
- ネットワーク基盤としての「広範囲をカバーする」性質を円運動で表現
- 安定したサービスであることを低速かつ予測可能な動きで表現
- 他のサービスの中心となる役割を、中央付近を周回する動きで表現
- 希少なリソースの重要性を、少数時の回復速度上昇で表現

### EBS (Elastic Block Store)

EBSは、EC2インスタンス用のブロックストレージとして、EC2の近くに留まろうとする動きをします。EC2との密接な関係性を視覚的に表現します。

**特徴:**
- **中速移動**: 最大速度が1.5に制限され、S3よりは速いがEC2ほど速くない
- **EC2への引力**: 最も近いEC2インスタンスに引き寄せられる
- **距離に応じた引力の強さ**:
  - 遠距離（200px以上）: 強い引力（0.2）
  - 中距離（100px〜200px）: 中程度の引力（0.1）
  - 近距離（50px〜100px）: 弱い引力（0.05）
  - 超近距離（50px未満）: 引力なし（すでに十分近い）
- **EC2の動きに影響**: EC2の近く（100px以内）にいる場合、EC2の動きに部分的に追従

**実装の意図:**
- EC2のブロックストレージとしての役割を、EC2を追いかける動きで表現
- EC2との密接な関係性を、近くにいる場合の動きの同期で表現
- ストレージとしての性質を、中程度の速度で表現

### Lambda (AWS Lambda)

Lambdaは、サーバーレスコンピューティングサービスとして、短時間の高速処理と休止状態を繰り返す特徴的な動きをします。イベント駆動型の性質を視覚的に表現します。

**特徴:**
- **状態ベースの動作**:
  - **アイドル状態**: ゆっくりと動き（最大速度0.8）、時々方向を変える（10%の確率）
  - **バースト状態**: 高速（3.0〜5.0）で直線移動、短時間（0.2〜0.3秒）のみ継続
  - **クールダウン状態**: 停止または非常に遅い動き、一定時間（0.5〜1秒）後に再びアイドル状態に
- **状態遷移**:
  - アイドル状態からバースト状態への移行確率: 3%
  - バースト状態からクールダウン状態への移行: バースト時間経過後
  - クールダウン状態からアイドル状態への移行: クールダウン時間経過後
- **IAMへの依存関係**: IAMが近くにない場合、Lambdaは引き寄せられる（バースト中を除く）

**実装の意図:**
- サーバーレスの「イベント駆動型」性質をランダムなバースト発生で表現
- 「短時間実行」の特性を短いバースト時間で表現
- 「オンデマンド」の特性をアイドル→バースト→クールダウンのサイクルで表現
- IAMロールへの依存関係を引力で表現（実行中は依存関係の影響を受けにくい）

### DynamoDB (Amazon DynamoDB)

DynamoDBは、NoSQLデータベースサービスとして、高速で予測可能なパフォーマンスと高いスケーラビリティが特徴です。頻繁な方向転換と速度変化で、分散データベースの特性を表現します。

**特徴:**
- **中高速移動**: 最大速度が2.0に設定され、データベースアクセスの高速性を表現
- **頻繁な方向転換**: 15%の確率で方向を変え（±90度）、ランダムアクセスパターンを表現
- **変動する速度**: 5%の確率で速度が急に変化し（加速または減速）、バーストキャパシティを表現
- **短距離バースト**: 2%の確率で短い距離を素早く移動し、読み取り/書き込みオペレーションを表現
- **画面全体への分散**: 画面端に近づくと中央方向への弱い引力が発生し、分散データベースの特性を表現
- **Lambdaへの依存関係**: Lambdaが近くにない場合、DynamoDBはLambdaに引き寄せられる傾向がある

**実装の意図:**
- NoSQLデータベースの「ランダムアクセス」特性を頻繁な方向転換で表現
- 「高いスケーラビリティ」を変動する速度で表現
- 「分散システム」の特性を画面全体に分散する傾向で表現
- 「バーストキャパシティ」を短距離バーストで表現
- 「サーバーレスアプリケーション」でのLambdaとの密接な連携を依存関係で表現

### API Gateway (Amazon API Gateway)

API Gatewayは、APIの作成、公開、管理、モニタリングを行うサービスとして、外部からのリクエストを受け付け、適切なバックエンドサービス（主にLambda）に転送する役割を持ちます。この「ゲートウェイ」としての特性を視覚的に表現します。

**特徴:**
- **状態ベースの動作**:
  - **パトロール状態**: 画面端付近を行き来する動き（入り口としての役割を表現）
  - **接続状態**: Lambdaに向かって高速で移動（APIリクエストの転送を表現）
  - **戻り状態**: 元の位置に戻る動き（APIレスポンスの返送を表現）
- **パトロールパターン**:
  - x軸またはy軸に沿って一定範囲を行き来
  - 範囲の端に到達すると方向転換
  - 時々パトロール軸を変更（x軸からy軸へ、またはその逆）
- **Lambdaとの連携**:
  - 2%の確率でランダムなLambdaを選択して接続状態に移行
  - Lambdaに到達すると相互作用を記録し、元の位置に戻る

**実装の意図:**
- 「ゲートウェイ」としての役割を画面端付近のパトロールで表現
- 「APIリクエスト処理」をLambdaへの接続と元の位置への戻りで表現
- 「リクエスト/レスポンスサイクル」を状態遷移で表現
- 「Lambdaとの連携」を視覚的な相互作用で表現

### RDS (Relational Database Service)

RDSは、リレーショナルデータベースサービスとして、安定性と信頼性を重視した動きをします。VPCへの依存関係と定期的なバックアップ処理を視覚的に表現します。

**特徴:**
- **軌道運動**: VPCまたは画面中央を中心とした円軌道上を移動
- **低速移動**: 最大速度が1.0に制限され、安定したデータベースを表現
- **滑らかな動き**: 目標位置に向かって徐々に移動し、急激な変化を避ける
- **VPCへの依存関係**: VPCが存在する場合、その周りを軌道運動
- **バックアップ状態**: 0.5%の確率でバックアップ状態に移行し、約2秒間速度が大幅に低下

**実装の意図:**
- リレーショナルデータベースの「構造化された」性質を規則的な軌道運動で表現
- 「安定性」と「信頼性」を低速かつ滑らかな動きで表現
- 「VPCへの依存関係」をVPCを中心とした軌道で表現
- 「定期的なバックアップ処理」を一時的な速度低下で表現

### CloudFront (Amazon CloudFront)

CloudFrontは、コンテンツ配信ネットワーク（CDN）サービスとして、S3などのオリジンから世界中のエッジロケーションにコンテンツを配信する役割を持ちます。広範囲をカバーする動きとS3との連携を視覚的に表現します。

**特徴:**
- **状態ベースの動作**:
  - **通常状態**: 広範囲をカバーする動き、S3の近くに留まろうとする傾向
  - **配信状態**: 画面の複数の場所（エッジロケーション）を高速で巡回し、その後S3に戻る
- **中高速移動**: 通常状態では1.8、配信状態では3.0の最大速度で、CDNの高速性を表現
- **S3への引力**:
  - 遠距離（200px以上）: 強い引力（0.1）
  - 中距離（100px〜200px）: 弱い引力（0.05）
- **配信パターン**:
  - S3の近くにいる場合、1%の確率で配信状態に移行
  - 画面の四隅付近にランダムに3〜5箇所の配信ポイントを生成
  - 各ポイントを高速で巡回した後、S3に戻る

**実装の意図:**
- CDNの「広範囲をカバーする」性質を画面全体を移動する動きで表現
- 「高速なコンテンツ配信」を高速移動で表現
- 「エッジロケーション」を画面の複数の場所への巡回で表現
- 「S3との連携」をS3への引力と相互作用で表現

### IAM (Identity and Access Management)

IAMは、AWSリソースへのアクセスを安全に管理するサービスとして、中央に位置する動きをします。アクセス管理の中心的な役割を視覚的に表現します。

**特徴:**
- **中央付近での軌道運動**: 画面中央付近を緩やかに周回する動き
- **低速移動**: 最大速度が1.2に制限され、安定したサービスを表現
- **強い中央引力**: 中央から200px以上離れると強い引力（0.1）で中央に引き戻される

**実装の意図:**
- 「アクセス管理の中心的役割」を画面中央付近での動きで表現
- 「安定したサービス」を低速かつ予測可能な動きで表現
- 「AWSアカウント全体への影響」を中央からの離れにくさで表現
